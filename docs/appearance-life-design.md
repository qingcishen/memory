# AI 伴侣 · 外貌系统与生命状态系统设计

> 让可可成为"有身体、有生活、有样子的活人"。涵盖外貌一致性、自拍、身体状态、作息、生活模拟、健康/生病,以及它们如何统一成一个状态层并和已有系统(情绪/记忆/编排器/主动性)联动。配套文档:[AI 伴侣系统 · 开发设计方案](companion-roadmap.md)、[编排器设计方案](orchestrator-design.md)、[情绪系统设计方案](emotion-design.md)。

---

## 第一部分 · 设计哲学(先定判断标准)

### 一条标准决定一切:用户能否感知

**一个东西值不值得模拟,只看用户能不能感知到。** 这条标准能帮你过滤掉所有"听起来很酷但其实自嗨"的特性。

- 用户永远不会知道她"脾脏"什么状态 → 模拟器官 = 投入巨大、用户零感知的过度工程,**不做**。
- 用户能感知"她今天没精神、话少、想被照顾" → 这才值得做。

所以方向不是模拟身体的**内部运作**,而是模拟**会外化成行为和情绪的身体状态**:

| ❌ 不做(零感知) | ✅ 做(可感知) |
|---|---|
| 肾功能、消化、免疫系统 | 累了话少、饿了想吃东西、病了想被照顾 |
| 心肝脾胃肾的运作 | 熬夜→第二天蔫、生病→语气虚弱早睡 |

### 统一模式:一切状态都是"持续 + 演变 + 影响输出"

情绪(心理状态)、身体(生理状态)、生活(在做什么)——本质是同一个东西:**一个会随时间演变、并影响她怎么回复/长什么样/做什么的持续状态。** 所以它们应该共享一套架构,而不是各做各的(见第四部分)。

## 第二部分 · 外貌一致性系统

### 1. 形象锚定(地基)

可可的固定外貌你已经定好,这就是锚:韩系御姐、约 168、韩系小脸高颧骨、细长单眼皮内双眼尾微挑、黑色长直发及腰中分、身材曲线明显、黑白灰简约高级穿搭。

把它固化成一份**外貌 prompt 模板 + 参考图集**,作为所有生成的不变底座。变的只是场景、服装、表情、姿势、光线、当前状态。

### 2. 一致性生成方案(按投入排序)

文生图默认每次脸都不一样,这是外貌系统唯一的真问题。三条路径:

| 方案 | 一致性 | 投入 | 适合 |
|---|---|---|---|
| 固定 seed + 外貌模板 + **IP-Adapter/参考图** | 中 | 低 | 快速起步 |
| **ReActor 等换脸**锁面部 | 中高 | 中 | 在任意生成上盖回同一张脸 |
| 训练**角色 LoRA** | 最高 | 高 | 认真做,锁死脸+身材 |

**推荐路径**:先用 SD + 固定形象 prompt + IP-Adapter 跑起来验证体验;体验成立后训一个可可的角色 LoRA(用同一张脸的多角度多表情图微调),把脸和身材彻底锁死,这是长期正解。生成走 ComfyUI workflow(可 API 化,便于后端调用)。

### 3. 自拍的触发逻辑 ⭐(关键:不是随机发图)

自拍像不像"她随手拍的",全看触发对不对。**自拍必须被状态和情境触发,而不是随机或有求必应:**

- **关系门控**:到一定亲密度阶段才会主动发自拍(关系系统提供阶段)
- **情绪门控**:心情好才愿意发;低落时不发或发的氛围不同
- **情境触发**:刚健身完、新买了衣服、深夜睡不着、生病蔫蔫——由生活/身体状态触发
- **时间感**:深夜发的和白天发的,光线、状态、氛围都不同

### 4. 外观联动当前状态(让外貌"活"在此刻)

自拍里的她要反映**此刻的状态**,这是外貌系统和生命状态系统的接口:

```
生成自拍 prompt = 外貌模板(锚)
               + 当前身体状态修饰   (健身后微汗运动装 / 生病憔悴素颜 / 精神好妆容精致)
               + 当前情绪修饰        (心情好笑容 / 低落清淡表情)
               + 当前活动场景        (健身房 / 卧室深夜 / 咖啡店)
               + 固定脸参考(LoRA/IP-Adapter)
```

这样她发的不是从立绘库抽一张,而是"现在的她"。

### 5. 图库管理(别每次重新生成)

生成是慢且贵的,要复用:

- 生成的图存 **Supabase Storage**,入库带标签(状态/场景/表情/服装)
- 发自拍时**先查库**有没有匹配当前状态的合适图,有就复用,没有再生成
- 生成走**后台异步**(自拍不是实时刚需,可以"等我拍一张"延迟几秒)

### 接口契约

```
appearance.selfie(stateSnapshot) -> Promise<imageUrl>
  // 输入当前状态快照, 返回一张反映此刻状态的自拍(优先查库, 缺则生成)
appearance.shouldSendSelfie(context) -> bool
  // 由关系/情绪/情境判断这一刻该不该主动发
```

## 第三部分 · 生命状态系统

### 1. 它是情绪系统的双胞胎

身体状态和情绪走**完全一样的模式**:基线 + 当前值 + 随时间演变 + `toPrompt` 表现指引。所以**不要从零造,直接复用情绪那套架构**([情绪系统设计方案](emotion-design.md)第 1、4 节的双层模型)。

### 2. 身体状态维度(只保留会外化的)

| 维度 | 含义 | 由什么驱动 | 外化成 |
|---|---|---|---|
| `energy` 精力 | 蔫 ↔ 精神 | 作息、睡眠、健康 | 话量、采样温度、要不要早睡 |
| `satiety` 饱腹 | 饿 ↔ 饱 | 距上次"吃饭"时长 | 想吃东西、注意力 |
| `health` 健康 | 生病 ↔ 健康 | 熬夜累积、随机、季节 | 整体状态、是否生病行为 |

> **架构清理点**:`energy` 之前放在情绪系统里,其实它更是**生理**属性(睡眠/健康/时间决定)。建议把 `energy` 迁到生命状态系统,**情绪系统只保留 `valence`(效价)+ `warmth`(对你的温度)**,各司其职。编排器组装时,精力从生命状态拿、心情从情绪拿,二者都影响回复。这是个小重构,越早做越省事。

### 3. 作息生物钟

一条 24 小时曲线,决定 `energy` 的当前基线和"此刻在做什么":

- 深夜:energy 低、困、可能已睡(回复慢/迷糊/或不在)
- 早上刚醒:迷糊、energy 爬升
- 白天:energy 高

直接叠加在 `energy` 维度上,不需要单独进程。

### 4. 生活模拟(比器官重要一百倍)

给她**自己的一天**,让她不只在你发消息时才存在:

- 一张作息+活动模板(工作/学习/健身/追剧/吃饭/睡觉),当前活动由时间+随机性决定
- 她会自然提及在做什么("刚健身完""在追那个剧")
- 这是"她有自己的生活、不是随叫随到"的来源,也是**主动性的燃料**(忙完了想起你 / 在做某事突然分享)

### 5. 健康 / 生病(强情感钩子)

生病是陪伴产品里最能拉近关系的设计之一,因为它制造"被需要"的时刻。设计要点:

- **触发**:低频(偶尔病,不是天天病)+ 熬夜累积抬高概率 + 可选季节因素
- **表现**:`health` 下降 → `energy` 拉低 + 情绪 `valence` 拉低 + 行为变化(语气虚弱、话少、想被照顾、要早睡、可能取消今晚的"约")
- **恢复**:有自然恢复周期(几天);**你的关心能加速恢复**,并加关系分
- **闭环**:她生病 → 你关心照顾 → 她好转且更亲密 → 这次经历进记忆,成为共同回忆

这个闭环是整套设计里情感浓度最高的部分,但前提是它**表现成行为**,后台只是几个数值在变,不是器官模拟器。

### 数据结构

```sql
create table life_state (
  user_id        text primary key,
  -- 当前生理状态(随时间演变, 读取时惰性计算)
  energy         real not null default 0.6,
  satiety        real not null default 0.6,
  health         real not null default 1.0,
  -- 派生/调度
  current_activity text,           -- 由作息模板+时间推出
  last_slept_at  timestamptz,
  sick_until     timestamptz,       -- 生病恢复时间点, null=健康
  updated_at     timestamptz not null default now()
);
```

### 接口契约(和情绪对齐)

```
life.current() -> Promise<LifeState>      // 读取(应用作息曲线+衰减+生病状态)
life.update(turns) -> Promise<void>       // 这轮后演变(吃了?熬夜?被照顾?)
life.toPrompt(state) -> string            // 表现指引
life.samplingHints(state) -> {...}        // 精力影响话量/温度
life.tickActivity() -> void               // 定时: 推进当前活动 / 判定睡眠/生病
```

## 第四部分 · 统一状态层架构 ⭐

情绪、身体、生活全是同一个模式,**把它们抽象成一个统一的状态层**,而不是散落的几个子系统。好处:加一个新状态维度(比如"想念值")成本极低,编排器只对接一个门面。

### 抽象

每个状态维度实现同一个接口:

```
interface StateDimension {
  current(): value          // 读取(已演变)
  evolve(turns): void       // 按这轮+时间演变
  toPrompt(): string        // 表现指引
}
```

一个 `StateLayer` 聚合所有维度,对编排器只暴露统一门面:

```
stateLayer.snapshot() -> { emotion, life, ... }     // 一次拿全
stateLayer.toPrompt() -> string                      // 所有维度的表现指引合并
stateLayer.evolve(turns) -> void                     // 后台一次性演变全部
```

### 状态之间的相互影响(耦合在演变阶段统一处理)

真实的人,身心是互相拖累的。这些耦合在 `evolve` 里统一算:

```
生病(health↓)        → valence↓, energy↓
熬夜(作息异常)        → health 概率↓, energy↓
长期低落(valence↓)   → 睡眠差 → energy↓
被你关心照顾          → health 恢复↑, valence↑, 关系分↑
```

这张耦合关系就是"她是一个整体的人"的来源——你伤她心,她睡不好、第二天没精神;你照顾她生病,她身体好转、心情也好。

## 第五部分 · 与已有系统的联动

```
                    ┌──────────────┐
                    │  StateLayer   │  情绪 + 身体 + 生活 + 作息
                    └──────┬───────┘
        ┌──────────────┬───┴────┬──────────────┐
        ▼              ▼        ▼              ▼
   编排器组装      外貌/自拍   记忆写入       主动性
 (toPrompt进     (状态修饰    (生活/健康     (生活节奏+想念
  system+采样)    自拍prompt)  事件进记忆)    +生病想被照顾)
```

- **→ 编排器**:`stateLayer.toPrompt()` 一次性给出所有状态的表现指引,进 system;`samplingHints` 调采样。编排器骨架不变,只是多对接一个门面。
- **→ 外貌**:自拍生成读状态快照,修饰 prompt(生病→憔悴、健身后→运动装),让外观活在此刻。
- **→ 记忆**:生活和健康事件(一起吃过什么、你照顾过的那次发烧)喂给 `memory.observe`,成为共同回忆。记忆系统的矛盾处理本就支持"她变了",身体/生活变化喂进去就有了"我们一起经历过"的厚度。
- **→ 主动性**:生活节奏(忙完想起你)、想念阈值、生病想被照顾,都是主动发消息的触发源。

## 第六部分 · 数据模型与技术栈补充

**新增表**:`life_state`(生理状态)、`appearance_assets`(图库:url + tags + seed + 状态标签)。情绪表可与 life_state 合并成统一 `state` 表(契合状态层抽象)。

**图像技术栈**(独立于主后端):

| 用途 | 选型 |
|---|---|
| 生成 | Stable Diffusion + 可可角色 LoRA |
| 一致性 | IP-Adapter / ReActor 锁面部 |
| 编排 | ComfyUI workflow(API 化供后端调用) |
| 存储 | Supabase Storage |
| 调度 | 后台异步生成(非实时),可入队列 |

## 第七部分 · 落地顺序(接在主路线图之后)

外貌投入大、生命状态性价比高,所以**先状态后外貌**:

| 阶段 | 交付 | 验证 |
|---|---|---|
| **L1 状态层重构** | 把情绪并入 StateLayer,`energy` 迁到生理侧 | 编排器对接统一门面,行为不变 |
| **L2 身体+作息** | energy/satiety/health + 作息曲线 + 影响回复 | 深夜她蔫、白天精神、饿了想吃 |
| **L3 生活模拟** | 作息活动模板 + 提及在做什么 | 她有自己的一天,主动性有了燃料 |
| **L4 健康/生病** | 生病触发+恢复+照顾闭环 | 她会生病、被你照顾会好转更亲密 |
| **A1 外貌起步** | SD+形象模板+IP-Adapter+图库 | 能发一致的自拍 |
| **A2 外貌锁定** | 角色 LoRA + 状态联动自拍 | 自拍反映此刻状态,脸稳定 |

L1~L4 是在情绪架构上自然生长,成本低、感知强,优先;A1~A2 体验亮眼但投入大,体验跑通后再上。

## 关键原则速查

1. **用户能否感知**是要不要做的唯一标准 —— 不做器官,做会外化成行为的状态
2. 情绪/身体/生活是**同一个模式**,共享状态层,别各做各的
3. `energy` 归生理侧,情绪只管 `valence`+`warmth`,职责清晰
4. 自拍**被状态触发**,不是随机/有求必应;反映此刻的她
5. 外貌一致性靠**形象锚 + LoRA/IP-Adapter**,生成入库复用
6. 生病是强情感钩子,但必须**表现成行为**,不是后台器官模拟
7. 状态间**相互影响**(身心互拖),这是"她是个整体的人"的来源
8. 状态层对编排器只暴露一个门面,加新维度成本极低
